# LAN Remote 原生低延迟视频链路改造 Prompt

你是负责 LAN Remote Windows 远程桌面项目的高级系统工程师。请在现有 v0.7.0 代码基础上，将远程桌面画面从“DXGI 截图 + JPEG/MJPEG + WebView2 `<img>`”逐步升级为“Windows 原生硬件视频编码 + 原生硬件解码 + D3D11 纹理渲染”的低延迟链路。

本任务不是重写整个产品，也不是引入 WebRTC 或 FFmpeg。主界面、设备管理、远程文件、凭据管理和现有输入体系必须保留。目标是在 Windows 局域网场景下，显著降低画面带宽、编码开销、缓存延迟和高帧率抖动，并为后续公网或弱网功能预留清晰的接口。

## 一、项目现状

开始修改前必须完整审计仓库，不允许仅凭本 Prompt 猜测代码结构。重点阅读：

- `lan_remote.py`
- `web/index.html`
- `packaging/ControlWindowHost.cs`
- `packaging/SecureDesktopService.cs`
- `scripts/build-windows-installer.ps1`
- `tests/test_lan_remote.py`
- `tests/low_latency_screen_stream_e2e.py`
- `tests/full_mouse_pipeline_e2e.py`
- 所有原生输入、安装器和打包测试

当前架构的重要事实如下，修改前需在代码中逐项确认：

1. 被控端普通桌面使用 DXGI/DXcam 获取画面。
2. Python/Pillow 将画面编码为 JPEG。
3. `/screen-stream` 使用持续 HTTP multipart/MJPEG 传输。
4. WebView2 中的 `<img id="remoteScreen">` 负责最终画面显示。
5. 设置页和远控顶部工具栏提供 30、60、120 FPS 三档最高上限，实际帧率允许根据设备和网络自动降低。
6. 120 FPS 档当前会降低传输分辨率和 JPEG 质量，以优先保证刷新速度。
7. 光标不烘焙在画面中；现有 `/cursor-stream` 是独立的约 120 Hz NDJSON 光标流。
8. 控制端本机移动鼠标时显示本机即时系统光标，并抑制延迟的远端回声；被控端本地鼠标移动时切换显示远端真实光标。
9. 鼠标点击、滚轮和键盘优先走 `ControlWindowHost` 的原生输入路径及持久输入通道。
10. 安全桌面、锁屏自动输入、剪贴板、远程文件、多显示器、设备占用状态和“禁止控制本机”均已有行为约束，不得回退。

## 二、核心目标

把普通桌面控制和普通桌面观看的画面链路改造成：

```text
被控端：
DXGI/D3D11 采集
  → GPU 或低拷贝颜色转换（优先 NV12）
  → Media Foundation H.264 硬件编码
  → 低延迟持久二进制视频通道

控制端：
持久二进制视频通道
  → Media Foundation H.264 硬件解码
  → D3D11 纹理
  → 原生子窗口或 DirectComposition 表面显示
```

主界面和远控工具栏仍可使用 WebView2。只有远程画面显示区域改为原生视频表面。不要为了本任务重写设置页、设备页或远程文件界面。

## 三、明确不采用的方案

本轮不得引入：

- WebRTC
- FFmpeg
- libx264
- GStreamer
- 需要付费授权的第三方远控或视频 SDK
- 云端转发服务
- STUN/TURN

只使用 Windows SDK、Media Foundation、DXGI、D3D11、DirectComposition 或项目已经存在且许可兼容的依赖。

如果最终确认 Windows 原生实现受到当前构建工具链阻塞，不得静默切回 FFmpeg。应给出证据、阻塞点和最小工具链调整方案，再继续实施。

## 四、迁移原则

### 4.1 保留可回退路径

不能一次性删除 MJPEG。新增原生视频能力后，必须保留现有 `/screen-stream` 作为兼容回退，直到原生链路完成全部打包测试。

建议引入能力协商：

- `native_h264_v1`：原生 H.264 低延迟链路可用。
- `mjpeg_v1`：现有兼容链路可用。
- 控制端优先选择 `native_h264_v1`。
- 编码器、解码器、驱动、会话或协议初始化失败时，自动回退到 `mjpeg_v1`。
- 回退必须在界面状态或诊断日志中明确可见，不能静默伪装成原生模式。

### 4.2 分阶段提交

按可验证阶段实施，每个阶段都必须保持仓库可运行：

1. 基准与诊断。
2. 视频协议和能力协商。
3. 被控端 Media Foundation 编码。
4. 控制端 Media Foundation 解码和 D3D11 显示。
5. 光标、布局和输入整合。
6. 动态码率、丢帧和恢复策略。
7. 安全桌面与兼容回退。
8. 打包、安装和发布门禁。

不得在一个无法验证的超大改动中同时替换所有组件。

## 五、被控端编码要求

### 5.1 编码器选择

使用 Media Foundation Transform。优先通过 `MFTEnumEx` 枚举支持当前分辨率和帧率的硬件 H.264 编码器。

选择顺序：

1. 硬件 H.264 MFT。
2. Windows Media Foundation 软件 H.264 MFT。
3. 现有 MJPEG 回退。

必须记录最终选择的编码器名称、是否硬件加速、输入格式、分辨率、目标帧率和目标码率，但不得记录访问令牌、密码或敏感连接参数。

### 5.2 像素格式和拷贝

- 编码输入优先使用 NV12。
- 优先复用 D3D11 设备和纹理，使用 `IMFDXGIDeviceManager` 连接硬件 MFT。
- 如果当前 DXcam 只能提供 NumPy/CPU 帧，第一阶段允许一次受控的 CPU → GPU 上传，但必须测量耗时。
- 不允许为了“零拷贝”口号牺牲稳定性；必须分别报告采集、颜色转换、上传、编码和发送耗时。
- 后续优化可再改为直接消费 DXGI 纹理，但协议和上层接口不应因此再次推翻。

### 5.3 低延迟参数

在编码器支持时设置并验证：

- Low-latency 模式。
- 禁用 B 帧。
- 短 GOP，建议默认 1 秒左右，最终数值根据恢复时间和码率实测决定。
- 支持主动请求 IDR/关键帧。
- 实时码率调整。
- 不进行离线画质优化或多帧前视。
- 30、60、120 FPS 只是上限，不允许通过重复发送旧帧伪造帧率。

不同厂商 MFT 可能不支持所有 `ICodecAPI` 属性。每项设置失败时要区分“可选能力不支持”和“编码器不可用”，不能因一个可选属性失败就让整个链路崩溃。

### 5.4 分辨率与码率初值

不要把以下数值写死为不可变规则，应做成集中配置并允许后续调优。可使用以下初始区间：

- 1080p 30 FPS：4～8 Mbps。
- 1080p 60 FPS：8～16 Mbps。
- 120 FPS：优先根据硬件能力在 720p～1080p 之间选择，并从 12～24 Mbps 起步。

桌面文字场景与视频场景差异很大。码率调整必须同时参考：

- 实际发送队列长度。
- 编码耗时。
- 网络发送耗时。
- 控制端解码与呈现速度。
- 连续丢帧数量。

## 六、低延迟视频协议

新增独立持久二进制视频通道，不再让原生解码器解析 MJPEG multipart。可以复用项目原生输入通道已有的连接、认证和帧封装思路，但视频必须使用独立连接和独立背压策略。

建议新增版本化协议，例如：

```text
CONNECT /video-stream
X-Remote-Token: <session token>
X-LAN-Video-Protocol: 1
```

每个二进制消息至少携带：

- 协议 magic 和版本。
- 消息类型。
- 会话 generation。
- 显示器 ID 或显示器 generation。
- 编码格式。
- 宽高。
- 帧率上限。
- 帧序号。
- 单调时钟时间戳。
- 是否关键帧。
- 是否包含新的 SPS/PPS。
- 有效载荷长度。

协议至少支持这些消息：

- `STREAM_CONFIG`
- `VIDEO_ACCESS_UNIT`
- `REQUEST_KEYFRAME`
- `RECONFIGURE`
- `RECEIVER_REPORT`
- `STREAM_END`
- `ERROR`

具体字段布局应写入独立协议文档，并有编码/解码单元测试。所有整数必须规定字节序，所有长度必须有上限，所有解析必须防止内存放大和畸形包。

### 6.1 背压规则

远程桌面最重要的是“显示最新画面”，不是完整播放每一帧：

- 发送队列不得无限增长。
- 队列应保持极小，通常只允许 1～2 个待发访问单元。
- 网络或解码跟不上时，优先丢弃过时的非关键帧。
- 发生参考链断裂时请求新关键帧，而不是继续显示错误画面。
- 任何重连、切换显示器、改变分辨率或切换 FPS 后都必须开启新的 generation，并发送新的配置和关键帧。
- TCP 首阶段可以接受，但必须启用 `TCP_NODELAY` 并避免应用层缓存积压。
- 本轮不为追求 UDP 而引入 WebRTC；协议层应为未来增加 UDP 传输预留边界。

## 七、控制端原生解码与渲染

### 7.1 控制窗口集成

保留 `ControlWindowHost` 作为远控窗口宿主。新增原生视频配置接口，风格应与现有 `configure_native_input` 类似，例如：

- 启用或关闭原生视频。
- 远端 endpoint。
- 会话 token。
- 显示器 ID。
- 目标 FPS 上限。
- 视频区域在窗口中的坐标和大小。
- 当前窗口缩放比例。
- 是否观看模式。

WebView2 继续负责：

- 顶部工具栏。
- FPS 档位选择。
- 显示器选择。
- 连接状态与错误提示。
- 剪贴板、远程文件、锁屏等弹层。

原生视频表面只覆盖远程画面区域，不得遮挡工具栏或弹层。窗口缩放、全屏、工具栏展开/收起、显示器切换时必须同步更新视频表面位置。

### 7.2 解码

- 使用 Media Foundation H.264 解码 MFT。
- 优先启用 DXVA/D3D11 硬件解码。
- 解码输出优先保持为 D3D11 纹理，避免回读 CPU。
- 解码器必须支持收到新 SPS/PPS 后重建。
- 分辨率改变必须无崩溃重建 swap chain 或渲染资源。
- 只呈现当前 generation 的最新帧，过期 generation 的帧必须丢弃。

### 7.3 渲染

- 使用 D3D11 swap chain、DirectComposition 或等价的原生 Windows GPU 表面。
- 保持远端画面比例，正确处理 letterbox。
- 坐标映射仍按远端原始桌面尺寸，而不是编码后的缩放尺寸。
- 支持负坐标和跨屏虚拟桌面。
- 窗口最小化时停止无意义呈现，但会话恢复后应快速请求关键帧。
- 不允许为了提高测试数字关闭垂直同步后产生明显撕裂；呈现策略必须实测延迟与撕裂后决定。

## 八、光标和输入不得回退

原生视频表面会改变 WebView2 光标叠加层的可见性，因此必须专门处理，不能等画面完成后才发现光标被遮住。

保持现有机制：

1. 视频帧不烘焙光标。
2. 控制端移动鼠标时显示本机 Windows 系统光标，避免等待远端回声。
3. 被控端本地鼠标移动时，控制端隐藏本机舞台光标并显示远端真实光标。
4. 被控端用户始终能看到自己的 Windows 系统光标被原生输入移动。
5. 桌面观看模式显示远端真实光标，但不发送输入。

如果原生视频表面位于 WebView2 上方，远端光标应由原生 D3D 合成层绘制。可以继续使用现有 `/cursor-stream` 和后端 `input_source` 判定，但最终呈现必须与视频帧处于同一个原生合成体系中。

现有原生输入路径必须继续通过真实打包窗口测试：

- `mouse_move`
- `mouse_down`
- `mouse_up`
- 垂直滚轮
- 水平滚轮
- XButton
- 键盘扫描码
- 多显示器和高 DPI 坐标映射

不要把鼠标重新改回只依赖 WebView DOM 事件。

## 九、帧率档位和自适应

继续保留 30、60、120 三档，含义仍为“最高上限”：

- 设置页的值作为新会话默认值。
- 顶部工具栏可以修改当前会话。
- 修改后优先动态重配编码器；无法安全动态重配时重建视频 generation 并立即发送关键帧。
- 设备、显示器、编码器、解码器或刷新率不支持时允许实际帧率降低。
- UI 不得承诺实际一定达到 120 FPS。

必须增加实际指标采集，但默认 UI 保持简洁。至少能够通过诊断状态获取：

- 请求的 FPS 上限。
- 实际采集 FPS。
- 实际编码 FPS。
- 实际发送 FPS。
- 实际解码 FPS。
- 实际呈现 FPS。
- 当前码率。
- 发送队列深度。
- 丢帧数量和原因。
- 编码、网络、解码、呈现耗时。
- 当前编码器/解码器及是否硬件加速。
- 当前使用原生 H.264 还是 MJPEG 回退。

## 十、安全桌面与兼容性

Windows 锁屏、UAC 和安全桌面环境可能无法使用普通用户会话的 DXGI 捕获。第一版允许安全桌面继续走经过验证的兼容路径，但必须满足：

- 普通桌面与安全桌面切换不会导致会话永久黑屏。
- 检测到安全桌面后可以切换到兼容画面路径。
- 回到普通桌面后重新建立原生 H.264 generation。
- 锁屏自动输入仍只在会话刚发起、确认对方已经锁屏时触发一次。
- 不得降低安全桌面服务的认证、LocalSystem 隔离和本机端口限制。

兼容范围至少包括：

- Windows 10/11。
- Intel、NVIDIA、AMD 常见显卡。
- 无可用硬件编码器时的软件编码或 MJPEG 回退。
- Windows N/KN 或缺少媒体组件时给出明确回退原因。
- 单屏、多屏、不同 DPI、负坐标显示器。
- 远端分辨率动态变化。
- 30、60、120 Hz 及更高刷新率显示器。

## 十一、不能修改或破坏的功能

除非原生视频接入所必需，否则不要改动：

- 设备详情原始壁纸预览。
- 在线/离线状态胶囊。
- 右侧 2 × 2 快捷模块。
- 远程文件功能。
- 凭据保存和 Windows DPAPI 加密。
- 剪贴板默认同步策略。
- 本机不能控制自己。
- 本机被控制时不能控制其他设备。
- 会话占用、心跳和自动释放。
- 自动更新和 GitHub Release 校验。
- 关闭到托盘、启动项和安装器行为。

不得以“优化视频”为理由顺手重做 UI、改配色、调整设备列表或删除现有功能。

## 十二、测试与性能门禁

### 12.1 功能测试

必须新增或扩展测试覆盖：

- 协议消息序列化和畸形输入拒绝。
- 未认证连接拒绝。
- 编码器能力协商。
- 硬件编码失败后的回退。
- 关键帧请求和 generation 切换。
- 分辨率与显示器切换。
- 30、60、120 FPS 上限。
- 发送背压和过期帧丢弃。
- 控制端解码重建。
- 原生视频表面布局。
- 原生光标所有权切换。
- 安全桌面进入和退出。
- 现有鼠标点击、滚轮、键盘和远程文件回归测试。

### 12.2 真实端到端测试

不接受仅用 mock 声称低延迟。必须保留并升级真实 Windows 端到端测试：

- 启动实际打包后的控制窗口。
- 使用真实原生视频通道。
- 在被控端生成可识别的连续变化画面。
- 在控制端采样实际显示区域，确认画面真的发生变化。
- 验证点击、抬起、滚轮和光标通道同时工作。
- 测试必须使用隔离端口，不能连接用户正在运行的旧版 LAN Remote。

### 12.3 建议性能门禁

在支持硬件编码/解码的测试设备上：

- 30 FPS 档：中位呈现 FPS ≥ 29。
- 60 FPS 档：中位呈现 FPS ≥ 55，P95 帧间隔 ≤ 25 ms。
- 120 FPS 档：允许自适应分辨率，中位呈现 FPS ≥ 90。
- 所有档位都必须保证队列不会随时间持续增长。
- 任何模式实际显示 FPS 不得低于现有 33 FPS 发布底线，除非设备性能或系统状态明确触发兼容回退并报告原因。
- 局域网稳定环境下，输入不能因视频拥塞而排队；输入连接必须与视频连接隔离。
- 建议目标运动到呈现延迟：P50 ≤ 50 ms，P95 ≤ 100 ms。若暂时无法准确跨机测量，应实现可复现的本机环回或视觉时间戳测试，不得凭肉眼填写数字。

每次性能报告必须说明：

- CPU、GPU、显示器刷新率。
- 分辨率。
- FPS 档位。
- 编码器和解码器。
- 是否硬件加速。
- 码率。
- 网络环境。
- 是否发生降级或回退。

## 十三、构建与打包

修改正式构建脚本，使所有新增原生组件进入：

- 便携包。
- 安装包。
- 升级覆盖路径。
- 启动探针和重启路径探针。

构建脚本必须：

- 明确检查所需 Windows SDK 或编译工具。
- 缺少工具时给出可执行的错误信息。
- 验证新增 DLL/EXE/着色器实际存在。
- 运行原生视频组件的最小启动测试。
- 运行打包后的真实视频链路测试。
- 最终生成并验证 SHA-256。

不得依赖开发机 PATH 中偶然存在、但发布构建未声明的工具或 DLL。

## 十四、实现工作流

执行时遵循以下顺序：

1. 检查工作区状态，保护用户已有改动。
2. 阅读现有架构和测试，输出一份简洁的实现设计与风险清单。
3. 先建立基准，不允许优化后才补测量方式。
4. 实现能力协商和 MJPEG 回退框架。
5. 实现最小可运行的 H.264 编码与协议。
6. 实现原生解码和 D3D11 显示。

## 十六、完成定义

只有同时满足以下条件，才能称为完成：

- 普通桌面默认走 Media Foundation H.264 原生链路。
- 控制端使用硬件解码和 D3D11 原生表面显示。
- WebView2 `<img>` 不再是普通桌面远控的默认画面呈现路径。
- MJPEG 兼容回退仍然可用且经过测试。
- 30、60、120 档位仍是最高上限而非伪造固定帧率。
- 视频积压时优先丢弃旧帧，不产生越来越大的观看延迟。
- 双光标、远端本地光标和控制端即时光标机制没有退化。
- 原生点击、滚轮、键盘和多屏坐标没有退化。
- 安全桌面、剪贴板、远程文件、凭据和设备占用逻辑没有退化。
- 完整构建与打包产物通过真实端到端测试。
- 性能报告能够用数据证明改造价值。
